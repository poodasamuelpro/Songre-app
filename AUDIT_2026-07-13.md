## P2 — Faisabilité : modification des durées d'éligibilité

> **Rappel** : Ce point est un audit de faisabilité uniquement. Aucune modification n'a été appliquée.

### 2.1 Localisation de la règle dans le code

**Valeurs actuelles** : Hommes = 60 jours, Femmes = 90 jours

La règle d'éligibilité est implémentée dans **2 fichiers distincts** :

#### Fichier 1 : `lib/models/models.dart`

```dart
// Ligne 223-227 — Calcul d'éligibilité
bool get estEligible {
  if (dernierDonDate == null) return true;
  final joursDepuis = DateTime.now().difference(dernierDonDate!).inDays;
  return genre == Genre.homme ? joursDepuis >= 60 : joursDepuis >= 90;
}

// Ligne 231-234 — Calcul date prochain don
DateTime? get prochainDonDate {
  if (dernierDonDate == null) return null;
  final delai = genre == Genre.homme ? 60 : 90;
  return dernierDonDate!.add(Duration(days: delai));
}
```

**Deux occurrences dans ce seul fichier** : `estEligible` (ligne 227) et `prochainDonDate` (ligne 233). Les deux doivent être modifiées simultanément.

#### Fichier 2 : `supabase/functions/retour-eligibilite-cron/index.ts`

```typescript
// Ligne 122 — Calcul identique côté backend
const delaiJours = profil.genre === "femme" ? 90 : 60;
```

Le cron pg_cron se déclenche quotidiennement à 08h00 UTC. Il récupère les donneurs dont `dernier_don_date + délai` tombe dans J+0 ou J+1 et envoie une notification de retour d'éligibilité.

**Note sur la plage de requête SQL** (lignes 88-93 du même fichier) :
```typescript
const dateMin60 = new Date(maintenant);
dateMin60.setDate(dateMin60.getDate() - 91); // max délai + 1 pour sécurité

const dateMax60 = new Date(maintenant);
dateMax60.setDate(dateMax60.getDate() - 59); // min délai - 1
```
Cette plage SQL capture les donneurs dont le dernier don date de 59 à 91 jours. **Si les délais passent à 90j/120j, ces constantes SQL devront aussi être mises à jour.**

#### Fichier 3 : `lib/screens/contact_screen.dart`

```dart
// Ligne 67 — Texte statique mentionnant les durées explicitement
r: 'Tout donneur éligible peut s\'inscrire. L\'éligibilité dépend du délai depuis le dernier don : 60 jours pour les hommes, 90 jours pour les femmes. L\'application vérifie automatiquement votre éligibilité.',
```
Ce texte est affiché dans une FAQ/section contact. Il contient **les durées en dur** et devra être mis à jour.

#### Fichier 4 : `README.md`

```markdown
| Homme | 60 jours |
| Femme | 90 jours |
```
Documentation — à mettre à jour pour cohérence.

### 2.2 Synthèse des fichiers à modifier

| Fichier | Ligne(s) | Type de modification | Risque |
|---------|----------|----------------------|--------|
| `lib/models/models.dart` | 227, 233 | `60 → 90`, `90 → 120` dans 2 expressions | Faible — côté calcul Dart uniquement |
| `supabase/functions/retour-eligibilite-cron/index.ts` | 89, 93, 122 | Plage SQL + délai logique | **Moyen** — redeployer l'EF |

### 2.3 Évaluation des risques

**1. Impact rétroactif sur les donneurs existants** :  
La règle `estEligible` est calculée **dynamiquement** à chaque appel, sans stocker la date de prochain don en base (seule `dernier_don_date` est persistée). Un changement de délai s'applique donc **immédiatement** à tous les utilisateurs existants dès le déploiement de l'APK :
- Un donneur homme ayant donné il y a 65 jours est actuellement éligible → avec 90 jours, il ne le sera plus pendant 25 jours supplémentaires.
- **Pas de migration de données nécessaire**, mais risque de surprise pour les utilisateurs.

**2. Impact sur le cron de notification** :  
Le cron `retour-eligibilite-cron` utilise une plage SQL hard-codée `(59 à 91 jours)`. Si les délais passent à 90/120 jours, la plage doit passer à `(89 à 121 jours)`. Sans ce changement, le cron continuera à envoyer des notifications aux bons moments (il filtre en mémoire), mais **la requête SQL initiale raterait les donneurs femmes** (filtre `<= 91 jours` alors qu'il faudrait `<= 121 jours`).

**3. Impact sur les notifications déjà planifiées** :  
Aucune notification n'est "planifiée" en avance — elles sont calculées le jour J par le cron. Pas de données à migrer.

### 2.4 Niveau de risque global et complexité

- **Complexité** : FAIBLE — 4 fichiers, modifications chirurgicales et localisées
- **Risque métier** : MOYEN — impact immédiat sur les utilisateurs actifs (recalcul de l'éligibilité)
- **Risque technique** : MOYEN — l'EF `retour-eligibilite-cron` doit être modifiée ET redéployée sur Supabase dans la même fenêtre que l'APK, sinon incohérence de 24h
- **Recommandation** : Coordonner le déploiement de l'APK et l'EF le même jour + communiquer aux utilisateurs le changement de règle

## P3 — Audit complet du système de notifications

### 3.1 Architecture du système

Le système de notifications SONGRE repose sur **deux canaux** :
1. **Email** : via `_shared/email.ts` (Brevo en priorité, Resend en fallback)
2. **FCM Push** : via `_shared/fcm.ts` → `FirebaseMessaging`
3. **In-app** : table `public.notifications_envoyees` lue par `SupabaseService.lireNotifications()`

**Flux d'une notification** :
```
Événement → Edge Function → notifierUtilisateur() → Email + FCM + INSERT notifications_envoyees
                                                                ↓
Flutter : _chargerNotificationsBackend() → lireNotifications() → _notifications list
```

**Note importante sur le contenu in-app** : Le backend stocke uniquement le `type` de notification (enum), pas de message texte. Le message in-app est généré côté Flutter par `NotificationSauve._messageDepuisType()` (dans `models.dart`). Cela signifie que le message est **statique et identique pour toutes les occurrences du même type** — il n'inclut pas le groupe sanguin, le prénom ni d'autres données dynamiques.

### 3.3 Anomalie #1 — `suppressionDemandee` : insertion conditionnelle in-app

**Problème identifié** : Dans `notifier.ts`, l'insertion en base `notifications_envoyees` n'est effectuée que si `emailSent || fcmSent` est vrai :

```typescript
// notifier.ts, lignes 230-245
if (!options.skipDbInsert && (result.emailSent || result.fcmSent)) {
  // INSERT notifications_envoyees
}
```

Pour `suppressionDemandee`, la notification est déclenchée via `_declencherNotificationSuppressionDemandee()` dans `supabase_service.dart`, qui appelle `mdp-modifie-auth` avec `{action: "suppression_demandee"}`. Ce mode utilise `notifierUtilisateur()` sans `skipDbInsert: true`, donc l'insertion en base se produit **seulement si l'email OU le FCM réussit**.

**Conséquence** : Si le serveur email et FCM sont tous deux indisponibles, la notification `suppressionDemandee` n'apparaîtra pas dans l'écran in-app. L'utilisateur ne saura pas si sa demande de suppression a été enregistrée.

**Sévérité** : FAIBLE (cas de défaillance infrastructure, non nominal).

### 3.4 Anomalie #2 — `bienvenue` : insert direct sans `notifierUtilisateur()`

**Code dans `bienvenue-auth/index.ts`** (lignes 125-145) :
```typescript
const emailSent = await envoyerEmailDirect(newUser.email, "bienvenue", { prenom });

// Persister la notification (non bloquant si erreur)
if (emailSent) {  // ← conditionnel sur emailSent uniquement
  const { error: insertError } = await adminClient
    .from("notifications_envoyees")
    .insert({ user_id: newUser.id, type: "bienvenue", lu: false });
}
```

**Problème** : `bienvenue-auth` n'utilise pas `notifierUtilisateur()` mais appelle directement `envoyerEmailDirect()`. La notification in-app n'est insérée qu'**après réception d'un email réussi**. Si l'email échoue, l'utilisateur ne voit aucune notification de bienvenue dans l'app.

De plus, `bienvenue-auth` est déclenchée par un **webhook Supabase Auth sur INSERT auth.users**. Si ce webhook n'est pas configuré dans le Dashboard Supabase, aucune notification de bienvenue ne sera envoyée du tout.

**Sévérité** : MOYENNE — dépendance configuration webhook externe, et pas de notification in-app si email échoue.

### 3.5 Compteur de notifications non lues (`notifNonLues`)

**Implémentation** :
```dart
// app_state.dart
int get notifNonLues => _notifications.where((n) => !n.lue).length;
```

Le compteur est calculé dynamiquement depuis la liste locale `_notifications`. La synchronisation est assurée par :
1. Chargement depuis le backend au démarrage (`_chargerNotificationsBackend()`)
2. Mise à jour optimiste locale immédiate lors du tap sur une notification
3. `PATCH notifications_envoyees?id=eq.$notifId` en fire-and-forget (`catchError` → silencieux)

**Risque** : Si le PATCH backend échoue (réseau), le compteur local affichera 0 mais la base de données conservera la notification comme non lue. Au prochain démarrage ou `actualiserNotifications()`, la notification réapparaîtra comme non lue → **compteur temporairement incohérent**.

**Sévérité** : FAIBLE — résolu au prochain rechargement.

## P5 — Faisabilité d'une version web sécurisée

> **Rappel** : Ce point est un audit de faisabilité uniquement. Aucune implémentation n'a été réalisée.

### 5.1 État actuel du support web

Le projet supporte déjà le build web (`flutter build web --release`). La version actuelle est **fonctionnelle mais non sécurisée** pour un usage en production avec de vraies données médicales.

**Ce qui fonctionne sur Web** :
- Navigation complète (GoRouter, ShellRoute)
- Formulaires (inscription, connexion, profil)
- Affichage des demandes et historique
- Consentements
- Saisie manuelle du token QR (fallback `kIsWeb`)
- `WebSecurityBanner` affiché pour avertir l'utilisateur

**Ce qui est dégradé ou absent sur Web** :

| Fonctionnalité | État actuel | Raison |
|----------------|-------------|--------|
| Stockage session (tokens JWT) | ⚠️ `localStorage` non sécurisé | `SecureStorageService` fait un fallback `SharedPreferences` sur `kIsWeb` |
| Scanner QR | ⚠️ Remplacé par saisie manuelle | `mobile_scanner` ne supporte pas Web |
| Notifications push FCM | ❌ Désactivé | `NotificationService.initialiser()` retourne immédiatement si `kIsWeb` |
| Largeur d'affichage | ⚠️ 430px max | `ConstrainedBox(maxWidth: 430)` dans `main.dart` |

### 5.2 Obstacles techniques et de sécurité

#### 5.2.1 Stockage de session (obstacle principal — sécurité)

**Problème** : Les tokens JWT (accessToken, refreshToken) sont stockés dans `localStorage` sur Web. N'importe quel script JavaScript s'exécutant sur la page peut les lire (attaque XSS).

**Solution standard** : Cookies `HttpOnly` + `SameSite=Strict` gérés par un serveur intermédiaire (reverse proxy ou serveur BFF — Backend For Frontend).

**Procédure** :
1. Déployer un serveur Node.js/Deno/Nginx devant l'application Flutter Web
2. Le serveur gère l'authentification Supabase côté serveur et émet un cookie HttpOnly
3. Le code Flutter Web appelle ce serveur au lieu de Supabase directement
4. **Impact code** : `SecureStorageService` doit être refactorisé pour Web (lire/écrire via des cookies HttpOnly au lieu de localStorage), ce qui nécessite une couche HTTP supplémentaire côté serveur

**Complexité** : ÉLEVÉE — nécessite un composant serveur supplémentaire (infrastructure + code serveur + refactorisation Flutter Web).

#### 5.2.2 Scanner QR (`mobile_scanner`)

**Problème** : `mobile_scanner: ^5.2.3` ne supporte pas la plateforme Web. Le fallback actuel (saisie manuelle du token) est déjà implémenté (`bool _showManual = kIsWeb`).

**Solution** : La Web Camera API (`html.window.navigator.mediaDevices.getUserMedia()`) peut être utilisée via `package:web` (Dart) ou `dart:html` + une bibliothèque de scan QR JavaScript (ex: `jsQR`). Cela nécessite un plugin custom ou la migration vers un package compatible Web.

**Complexité** : MOYENNE — faisable mais nécessite un plugin custom ou une intégration JS.

#### 5.2.3 Notifications push FCM

**Problème** : Firebase Cloud Messaging pour Web utilise les **Service Workers** et l'API Web Push. `firebase_messaging: 15.1.3` supporte techniquement Web, mais nécessite :
1. Un fichier `firebase-messaging-sw.js` dans `web/`
2. La configuration Firebase Web App (apiKey, projectId, etc.) dans `web/index.html`
3. Un token FCM Web (différent du token Android)

Le code actuel fait `if (kIsWeb) return;` dans `NotificationService.initialiser()`, ce qui désactive complètement le FCM Web.

**Solution** : Configurer Firebase Web dans `firebase_options.dart` et adapter `NotificationService` pour gérer le cas Web.

**Complexité** : MOYENNE — 2-3 jours de travail si Firebase Web App déjà créée.

#### 5.2.4 Contrainte de largeur 430px

**Problème** : La contrainte `ConstrainedBox(maxWidth: 430)` dans `main.dart` force une mise en page mobile sur Web. Sur un écran desktop 1920px, l'application apparaît centrée en colonne étroite.

```dart
// lib/main.dart, ligne 83-85
if (kIsWeb) Center(
  child: ConstrainedBox(
    constraints: const BoxConstraints(maxWidth: 430),
```

**Analyse** : Pour une application destinée à un usage mobile avec conversion mobile → web, cette approche est correcte et courante (pattern "mobile-first web"). Elle évite une refactorisation complète de toutes les pages.

Pour une vraie expérience web desktop, une mise en page responsive complète (grilles à 2-3 colonnes, sidebar navigation) serait nécessaire — ce qui représente une refactorisation majeure de l'UI.

**Complexité** : FAIBLE à maintenir tel quel (acceptable), TRÈS ÉLEVÉE pour une vraie adaptation responsive.

### 5.3 Procédure concrète vers une version web sécurisée

**Étape 1 — Infrastructure serveur BFF (Backend For Frontend)** :
- Déployer un serveur Node.js/Deno sur un hébergeur (Vercel, Cloudflare Workers, VPS)
- Implémenter un endpoint `/auth/login` qui appelle `supabase.auth.signIn()` côté serveur et set un cookie `HttpOnly`
- Implémenter un endpoint `/auth/logout` qui supprime le cookie
- Implémenter un middleware qui lit le cookie et injecte le JWT Supabase pour les requêtes API

**Étape 2 — Refactorisation `SecureStorageService`** :
- Sur Web, remplacer l'écriture/lecture localStorage par des appels HTTP au serveur BFF
- Le serveur gère le cookie HttpOnly, le client Flutter Web ne touche jamais le JWT

**Étape 3 — FCM Web** (optionnel si push mobile suffit) :
- Créer une Firebase Web App dans Firebase Console
- Ajouter la configuration dans `web/index.html` et `firebase_options.dart`
- Créer `web/firebase-messaging-sw.js`
- Adapter `NotificationService` pour ne plus faire `if (kIsWeb) return`

**Étape 4 — Scanner QR Web** (optionnel si saisie manuelle acceptée) :
- Intégrer `jsQR` via `dart:js_interop` ou un package Flutter Web compatible
- Remplacer le fallback manuel par une vraie webcam

**Étape 5 — Tests de sécurité** :
- Audit XSS (vérifier que les cookies HttpOnly ne sont pas accessibles depuis JS)
- Test CSRF (ajouter un header anti-CSRF si le BFF gère des mutations)
- Test de l'expiration de session
- Vérification HTTPS obligatoire (les cookies SameSite=Strict ne fonctionnent pas en HTTP)

### 5.4 Niveau de difficulté global

| Composant | Difficulté | Estimation |
|-----------|------------|------------|
| Serveur BFF + cookies HttpOnly | ÉLEVÉE | 2-3 semaines |
| Refactorisation SecureStorageService Web | MOYENNE | 3-5 jours |
| FCM Web | MOYENNE | 2-3 jours |
| Scanner QR Web | FAIBLE-MOYENNE | 1-2 jours |
| Tests de sécurité | MOYENNE | 1 semaine |
| **Total** | **ÉLEVÉE** | **~1,5 mois** |

**Conclusion P5** : Une version web sécurisée est techniquement faisable à partir du code Flutter existant, mais nécessite un investissement significatif. Le principal obstacle est l'authentification sécurisée (cookies HttpOnly via BFF), qui est une refactorisation architecturale, pas un simple correctif. La version actuelle convient pour la démonstration. Pour un déploiement web en production avec de vraies données médicales au Burkina Faso, l'étape BFF est non négociable.
